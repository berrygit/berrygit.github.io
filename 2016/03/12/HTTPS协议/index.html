The First Few Milliseconds of an HTTPS Connection

关于HTTPS协议非常详细的一篇文章，有兴趣可以详细读读，本文只对HTTPS流程做总结。

必要性

http通信通常有三种风险：

  1. 窃听风险。
  2. 篡改风险，如网页中被植入的广告。
  3. 冒充风险。

于是SSL/TSL相应就采取了如下3种措施来防范这些风险：

  1. 所有信息加密传播，防止窃听。
  2. 具有校验机制，一旦被篡改通信双方会立刻发现。
  3. 配备证书机制，防止身份被冒充。

原理

要解释HTTPS协议，首先看一张图，可谓一图胜千言！

在传统的HTTP与TCP层之间加上TLS/SSL协议，该协议负责将明文报文加密后交给TCP层传输然后接收端解密后给HTTP层，这样在网络中传输都是密文，从而提高安全性。该协议属于会话层，对应用层透明。SSL（Secure Cockets Layer 安全套接层）由网景公司设计，IETF把SSL标准化后改名为TLS（Transport Layer Security 传输层安全协议），所以这两者其实是同一个东西的不同阶段。

流程

首先，浏览器访问https协议的网址会默认访问443端口。在正常的HTTPS协议传输前，会使用TLS/SSL协议与服务器协商待传输HTTPS协议报文的加密方式及具体使用SSL/TLS协议的版本号，使得两方都有对称加密算法的秘钥。一旦协商完成，以后的HTTPS报文便直接加密后交给TCP协议传输。

Client Hello

浏览器向服务器发起握手请求，在这一阶段发送的报文主要是浏览器方与服务器方确认SSL/TLS协议的版本号以及浏览器方所支持的加密方式的组合，提交给服务端，服务端选择一种加密方式的组合。这里的加密方式一共有三个，首先是对称加密用来加解密以后用来传输HTTP报文的加密算法；其次是用来加密提交给服务器的对称加密的秘钥的非对称加密算法；最后是用来校验传输过程中数据完整性的哈希算法。由于非对称加密需要较大的计算开销，所以使用对称加密方式加密待传输的报文，用非对称加密来加密较短的对称加密的秘钥，即保证安全性，也兼顾性能开销。

Server Hello

服务端回复浏览器的握手请求，返回对SSL/TLS协议的版本号的确认，以及一种加密方式的组合，同时返回sessionID可以让下次有HTTPS传输时不用再次做昂贵的握手，服务器方也会把自己的证书发给浏览器方。

证书检验

对于服务端发来的证书我们去校验它的合法性，一是要保证发来的证书在传输过程中没有被人串改，另外也要保证发来的证书是可信的。那么如何验证？

这里要涉及到CA证书的信任链以及数字签名。首先对于一个陌生的网站我们无从知晓它是否可信，那么我们就去找一些值得信任的机构，如果陌生网站想要被人信任就去信任机构注册，由信任机构去保证这些网站的可信。信任机构在审核网站的注册请求后，会给这些网站发布一个凭证，这个凭证就是证书。有了证书后，这个网站就是可以被信任的，同时他可以发布自己的信任的网站了，这就是个信任链。

每次我们使用HTTPS协议的时候，服务器会把这个证书发给我们。那么下面的问题就是如何确定服务器发来的证书是可信的，而不是任意一个人随意生成的。这里涉及到一个数字签名的问题，首先作为信任机构它们生成了自己的根证书，这些根证书在浏览器软件的发布公司默认的放置一些，也可以自己在浏览器中安装（如12306），根证书中有信任机构公钥，这样我们在浏览器端已经有了信任机构的公钥。服务器端如果是正规的证书，是不能自己生成的，它需要网站的相关人员去信任机构注册，信任机构根据你提交的信息，生成证书并对证书做一个哈希算法，生成证书的指纹，同时它用自己的私钥对指纹加密生成签名。
神奇的事情就在这里发生，注册好的证书是一个文件，有信任机构的私钥生成的签名，同时证书中记载着哈希算法和签名算法的具体信息。当用户访问网站，网站把证书交给浏览器，浏览器看到证书里的签名算法描述，先用哈希算法对证书生成指纹，同时根据证书中记载的信任机构信息，找到信任机构的公钥，对签名用公钥解密，这样如果解密后的签名与我们自己计算的签名值一致。我们可以确定：1.该证书不是某个人随意生成的，而是信任机构审核过的，证书里的信息真实可信；2.证书在传输过程中没有被人串改。

到这里我们可以确信，网站发给我们的证书是可信，我们就这样拿到了网站的公钥，在浏览器中我们可以看到一把绿色的锁头。

注意
需要注意的是，一般证书的都有有效期，一般为一年。因为根据公钥我们有可能计算出对应的私钥，如果一个公钥用得越久被破解的可能性就越大。

秘钥协商

一般需要使用一个随机数做为对称加密的秘钥，但是生成一个完全意义上的随机数是不可能的，而如果随机数不是随机的那么秘钥就并不安全。为了提高秘钥的安全性，SSL使用如下办法：

  1. Client Hello中生成一个随机数。
  2. Server Hello中生成一个随机数。
  3. 浏览器端随机生成48字节的Pre-Master Secret。
  4. 浏览器端根据上面的三个随机数生成最后的主秘钥Master Secret。
  5. 浏览器端将Pre-Master Secret做哈希以及用公钥加密后，传给服务器端。
  6. 服务器端解密，并做完整性校验，得到Pre-Master Secret，并按照相同规则生成Master Secret。
  7. 这样两方都有了主秘钥Master Secret。

秘钥协商结束后，所有的http报文会被SSL/TLS加密后交给TCP传输。可以理解为原来的http报文，变为TLS的版本号+加密后的报文+校验码。
